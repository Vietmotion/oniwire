# DEV LOG - 2026-04-19

## Summary

Today we documented and fixed a runtime conflict where motion behavior looked broken even though node logic was valid in source.

Result:
- `Motion - Act` runtime issue was traced to cache/version drift, not node graph logic.
- Script query versions were updated in the HTML entry.
- Live entry (`index.html`) was synced and deployed.
- Rendering knowledge base was updated with a reusable incident pattern.

## Conflict Observed

- User symptom: `Pen -> Motion - Act` and `Text -> Motion - Act` appeared to do nothing.
- Initial risk: could be interpreted as render pipeline regression.

## Root Cause

- HTML script query versions were stale versus actual JS change dates.
- Browser kept serving older cached `scripts/nodes/motion.js` and `scripts/export.js`.
- Runtime behavior therefore did not match latest repository code.

## Fix Applied

1. Updated cache-busting script query versions in the app HTML entry:
   - `scripts/nodes/motion.js?v=20260327a`
   - `scripts/export.js?v=20260329a`
2. Synced latest app file into `index.html`.
3. Pushed deploy commits to `main`.

## Why This Matters For Rendering

This is a high-frequency false-regression pattern:
- render nodes can appear broken,
- while the real failure is stale runtime assets.

Treat this as a first-pass check before deep renderer rewrites.

## Prevention Rule Added

Before diagnosing render logic deeply:
1. Check script query token freshness.
2. Hard refresh and private-window verify.
3. Confirm `index.html` was synced before push.

## Documentation Follow-up

- Added incident to:
  - `docs/RENDERING_KNOWLEDGE_LOG.md`

This keeps the case in the long-lived rendering experience library for future debugging.
