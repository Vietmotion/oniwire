# DEV LOG - 2026-04-10

## Summary

Goal today: speed up MP4 export for heavy projects (hundreds of nodes).

Result:
- Replaced frame-by-frame DOM snapshot rendering with real-time on-screen capture for MP4 export.
- Export now records the live preview region from the shared browser tab instead of re-evaluating full scene rendering for every frame.
- Applied the same fix to both `oniwire_v0_2_2.html` and `index.html` to keep runtime parity.

## Problem Observed

- Existing MP4 export loop used per-frame capture logic (`freezeAnimationsAtTime` + snapshot capture path) for all frames.
- On large node graphs this became very slow because each export frame triggered heavy render/capture work.

## What Changed

### 1. MP4 export switched to real-time screen capture path

In both app files:
- `oniwire_v0_2_2.html`
- `index.html`

`exportMp4(name)` now:
- Uses `navigator.mediaDevices.getDisplayMedia()` to capture the current browser tab stream.
- Uses a hidden `<video>` fed by that stream as the frame source.
- Crops the captured frame to the preview (`renderRoot`) bounds and encodes to H.264 MP4 via WebCodecs + `mp4-muxer`.
- Keeps existing progress overlay and cancel flow.

### 2. UX / robustness updates

- Added explicit browser capability check for `getDisplayMedia`.
- Added permission-denied toast guidance for capture sharing.
- Stops capture tracks and clears video source in `finally` for clean teardown.

## Expected Impact

- Large-project export should be much faster in practice because it records what is already rendered on-screen in real time.
- Tradeoff: export duration is now closer to real-time recording duration instead of offline frame-by-frame rendering time.

## Manual Verification Checklist

1. Open a heavy project (100+ nodes).
2. Click `Export MP4`.
3. In browser share picker, choose the current app tab.
4. Confirm resulting MP4 captures the preview area correctly.
5. Compare export time vs previous implementation.

## Notes / Pending

- Capture accuracy depends on selecting the current browser tab in the share prompt.
- If user selects a different surface (window/monitor), crop alignment may differ.
- If needed next, add a toggle in UI for two modes:
  - `Realtime Capture (fast)`
  - `Frame Accurate (slow)`

## Reliability Hardening (Later Update, same day)

Applied follow-up fixes after deep export-path audit.

### 1) Animated mask correctness in MP4

- Before: mask canvas was built once before frame loop and reused for all frames.
- Risk: moving/animated masks could drift from frame content.
- Now: if mask motion is detected, mask canvas is rebuilt per frame before capture.

### 2) MP4 fps sync

- Before: MP4 export used hardcoded `30` fps.
- Now: MP4 uses project/output fps when present, with safe clamp `1..60`, default `30`.

### 3) Local-first dependency loading for export libs

- Added local vendor copies:
  - `vendor/html2canvas.min.js`
  - `vendor/mp4-muxer.min.js`
- Export loader now resolves in this order:
  1. local vendor file
  2. jsdelivr CDN
  3. unpkg CDN

This reduces export failures when one network source is blocked/unavailable.

### 4) UX wording cleanup

- MP4 progress note now accurately reflects current mode:
  - `Frame-by-frame MP4 render in progress.`

Updated files:
- `oniwire_v0_2_2.html`
- `index.html`
- `scripts/nodes/output.js`
- `vendor/html2canvas.min.js`
- `vendor/mp4-muxer.min.js`
